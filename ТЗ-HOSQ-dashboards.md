# Техническое задание — Дашборды фонда HOSQ

**Версия:** 1.0 · **Дата:** 2026-06-29 · **Источник данных:** `hosq_project_profile_v7_enriched.xlsx` (47 проектов, статус draft)
**Хостинг:** Render.com (статика/Node, через `onrender-deploy`) · **Доступ:** публичный, без PII

> Документ собран пятью экспертами (стратегия/стейкхолдеры, BI/данные, инфодизайн, M&E культурного импакта, веб-инжиниринг), работавшими автономно и сведёнными в общее решение. Решения по 4 ключевым развилкам зафиксированы заказчиком.

---

## 1. Цель и принцип

Набор онлайн-дашбордов, который отвечает на частые вопросы ключевых стейкхолдеров HOSQ и даёт основания для управленческих решений (финансирование, ресурсы, стратегия, отчётность донорам).

**Главный принцип — честность данных.** Данные черновые, с пропусками. Дашборд показывает только то, что подтверждено, с указанием охвата («X из 47») и даты. Пустое поле никогда не превращается в «0». Фреймворк A–J показывается как **покрытие доказательствами**, а не как выдуманные баллы.

## 2. Ключевые стейкхолдеры → Jobs To Be Done

| Стейкхолдер | Главная работа (JTBD) | Решение |
|---|---|---|
| **Доноры / грантодатели** | «Когда я оцениваю заявку HOSQ под критерии фонда, я хочу видеть сопоставимые доказательства импакта, чтобы решить, давать ли грант €30–50k.» | финансировать / продлить / сколько |
| **Борд / руководство фонда** | «Когда я планирую следующий спринт, я хочу видеть, какой стрим/тип даёт больше импакта на доллар, чтобы решить, куда вложить деньги.» | распределение бюджета и команды |
| **Кураторская команда** | «Когда я планирую новый проект, я хочу бенчмарки прошлых проектов (бюджет, команда, длительность), чтобы правильно его ресурсить.» | операционные решения |
| **Партнёры** | «Когда я решаю продлевать ли сотрудничество, я хочу видеть охват и сеть HOSQ, чтобы обосновать это у себя.» | продлить / углубить |
| **Публика / пресса** | «Когда я встречаю HOSQ, я хочу понять масштаб и разнообразие, чтобы прийти / следить / написать.» | внимание, амплификация |

Полная карта JTBD по каждому стейкхолдеру (2–4 работы + частые вопросы) — в приложении A.

## 3. Три дашборда (продукта)

Все три — один связанный сайт, публичный, контакты вычищены.

### Д1 — «Портфель» (`index.html`) — борд + доноры + публика
- **Hero KPI-полоса:** # проектов, суммарный бюджет (по N из 47), офлайн-охват, локальные партнёры, языки.
- **Графики:** бюджет по стримам (бар), проекты по типам (бар), по годам (бар), покрытие доказательствами A–J (бар, приоритетные векторы коралловые).
- **Фильтруемая сетка проектов** (стрим / тип / год) → клик ведёт в детальный профиль.

### Д2 — «Импакт A–J» (`impact.html`) — доноры + оценщики
- **Карта покрытия** проекты × 10 векторов (heatmap, цвет = доля заполненных доказательств).
- **Карточки 10 векторов** с баром покрытия и бейджем «приоритет донора» (G, A, D).
- Жёсткий дисклеймер: «покрытие доказательствами, не балл качества».

### Д3 — «Витрина» (`showcase.html`) — публика + пресса + партнёры
- Брендовая фильтруемая галерея проектов.
- **Кейс флагмана Notations Lab** (87 человек, дисциплины, партнёры, события) — явно помечен как один кейс, НЕ портфельные цифры.

### Профиль проекта (`project.html?slug=…`)
KPI проекта, радар покрытия A–J, нарративные доказательства, ссылки. Индивидуально шарится.

**Система JTBD-плейсхолдеров:** каждая карточка несёт строку `JOB: Когда [стейкхолдер] хочет [работу], это показывает [ответ]` + источник (поле) + дата + пустое состояние.

## 4. Критические факты о данных (определяют дизайн)

1. **events / team / partners НЕ соединяются с 47 проектами** — у всех `project_id="nl"`. Это одна программа Notations Lab. → Никаких портфельных агрегатов из этих таблиц; только отдельный кейс.
2. **Системно пустые поля:** `jobs_created`/`jobs_paid` (0 везде), `partners_international` (0 везде), онлайн-охват (1 значение = 2 млн, флагман). → Трактуем как «нет данных», флагуем, не публикуем как «0». Онлайн-2млн — в карантин (только кейс Notations Lab, помечен «self-reported, не верифицировано»).
3. **A–J не имеет числовых баллов** — 40 субиндексов привязаны к полям-доказательствам. → Показываем долю заполненных полей (Evidence Coverage), не баллы.
4. **Покрытие по векторам (факт):** A 52% · E/G/J ~28–30% · F 28% · H 21% · D 16% · C 15% · B/I ~7%.
5. **Охват KPI:** бюджет 27/47 ($375k) · офлайн-охват 30/47 (3 189) · участники 5/47 · локальные партнёры 40/47.

**Приоритет донора (зафиксирован):** Интернациональность **G** + Культурная витальность **A** + Художественное качество **D** — выносятся вперёд.

## 5. Модель данных и пайплайн

```
xlsx (6 листов) ──[scripts/build_data.py]──► data/*.json (версионируется в git) ──► статический сайт (fetch)
```

**Сущности:** project (PK `project_id`, 47) · metric vector/subindex (A–J, 40) · поле→субиндекс (из `schema.hosq_metric_codes`). Реляционные events/team/partners — отдельным namespace `notations_lab`.

**Выходные файлы:**
- `manifest.json` — версия, дата, # проектов, honesty-note, флаг pii_stripped.
- `projects.json` — 47 кратких записей (фасеты, KPI, vector_coverage, flags).
- `projects/{slug}.json` — полный профиль с нарративами и subindex_evidence.
- `aggregates.json` — портфельные суммы с знаменателями + vector_coverage + список blocked_metrics.
- `metrics_legend.json` — 10 векторов и субиндексы.
- `notations_lab.json` — кейс флагмана (PII вычищен).
- `build_report.txt` — отчёт о покрытии/пропусках для публикатора.

**Правила обработки (в скрипте):** `GAP*`/пусто → null; `0` в jobs_*/partners_international → null + флаг; онлайн ≥1M → карантин; PII (email, telegram, contact_person, contact_email) → удаляется.

## 6. Технологии

| Слой | Решение |
|---|---|
| Хостинг | Render (Node static `server.js`), деплой через `onrender-deploy` (org symbioticart) |
| Фронт | Чистый ES-modules JS + общий модуль `src/common.js`. Без фреймворка (47 записей) |
| Графики | **ECharts** (radar + heatmap из коробки, тема под бренд) |
| Данные | предсобранный статический JSON, fetch на клиенте |
| Стиль | `styles/app.css` — токены бренда HOSQ (cream #EBE2D6, coral, violet, green, blue; Cera Pro; пилюли 999px; без теней) |
| Шрифты | локальные Cera Pro WOFF, `font-display:swap` |

**Нефункциональные:** payload < ~100 КБ на страницу; respansive (grid auto-fit); a11y (семантика, `aria-pressed` фильтры, не только цвет); пропуски → «—», секция без данных скрывается; OG-теги для шаринга.

## 7. Обновление данных (для не-разработчика) — `REFRESH.md`

1. Положить новый `hosq_project_profile_v7_enriched.xlsx` в `source/`.
2. `npm run data` (перегенерит `data/*.json`).
3. Глянуть `data/build_report.txt` на предмет MISSING.
4. `git add -A && git commit -m "data refresh" && git push` → Render задеплоит за ~1 мин.
